Project Rulebook (AI Handoff)

This file is the source-of-truth guide for AI agents working in the current game module.
Use this to understand structure, runtime flow, editor flow, style, and constraints.

========================================
1) Scope and Structure
========================================

Scope:
- Primary module: Assets/<ModuleName> (current: Assets/Mahjong)
- Script roots:
  - Assets/<ModuleName>/Scripts/Core
  - Assets/<ModuleName>/Scripts/Development
  - Assets/<ModuleName>/Scripts/Editor

High-level folders:
- Animations, Materials, Prefabs, Shaders, Sounds, Sprites, Textures, UI, VFX
- Databases (levels, audio, tile database assets)

Script architecture:
- Core: pure data + enums
  - LevelData, LayerData, MissionData, TileData, PlayerData
- Development: runtime systems
  - Bootstrap, Board, Tile, Input, Rules, StateMachine, UI, Pool, LoadSave, VFX
- Editor/MapMaker: level authoring tools (paint/move/save/load/validate/shuffle)

Important note:
- No .asmdef exists under the current module folder. Scripts compile into the default assemblies.

========================================
2) Runtime Startup and Scene Flow
========================================

Scene flow:
1. Loading scene runs LoadingBar.
2. LoadingBar additively loads gameplay scene (default "GamePlay"), fades out loading UI, invokes:
   - EventManager.OnGameplayRevealCompleted
3. Gameplay systems continue reveal/entrance flow from this event.

Main runtime entrypoint:
- Scene entrypoint class (current: GameBootstrap at Assets/Mahjong/Scripts/Development/GameBootstrap.cs)

GameBootstrap flow:
1. Awake:
   - Set target frame rate.
   - Init PlayerDataService.
   - Resolve requested level (default hardcoded fallback is level 12).
   - Load LevelData via MessagePackRepository (Addressables key: Level_<id>).
2. StartLevel wiring:
   - board.Init(levelData, boardState)
   - boosterController.Init(boardState, inputController)
   - selectLogic = new SelectLogic(boardState, tilePool)
   - inputController.Init(selectLogic, boardState)
   - create GameContext and GameStateMachine
3. Start:
   - stateMachine.Activate()

Session persistence:
- Snapshots are written to persistentDataPath/game_session.json.
- Save is triggered on pause/focus lost/destroy/quit when resumable.
- Resume loading path exists but is currently disabled in Init() (commented call to TryLoadGameplaySession).

========================================
3) State Machine Flow
========================================

Implementation:
- Stateless package
- State enums:
  - Loaded, Tutorial, Started, Paused, Restart, Won, Lost

Transitions:
- Loaded:
  - CurrentLevel <= 2 => Tutorial
  - else => Started
- Tutorial:
  - FinishTutorial => Started
- Started:
  - Pause => Paused
  - Win => Won
  - Lose => Lost
- Paused:
  - Resume => Started
  - Restart => Restart
- Lost:
  - Revive => Started
- Restart:
  - reload active scene
- Won:
  - claim reward then level up and reload active scene

StartedState responsibilities:
- Mission tracking and mission UI updates
- Combo tracking and combo audio
- Countdown timer and time-up lose
- Win possibility checks and add-moves lose
- Result transition delay before Win/Lose trigger fire

========================================
4) Board, Tile, and Rule Flow
========================================

Board creation:
- Board runtime controller (current: MahjongBoard) builds runtime tile objects from LevelData.Layout.
- Uses TilePool.
- Applies TileData properties (position, layer, flip, kind, capacity).
- Resolves icon/text from TileDatabase by TileInfoID.

Tile model:
- Tile runtime entity (current: MahjongTile) state:
  - TileKind (Normal/Root)
  - RuntimeState (Normal/Selected/Removed)
  - FlipState
  - Capacity (for root tiles)

Selection and match:
- SelectLogic owns click-to-select and match behavior.
- Pairing rule: same Category.
- Root tile behavior:
  - match decrements capacity
  - remove only when capacity <= 0

Free-tile rule (current implementation: ClassicMahjongRule):
- Tile is NOT free if:
  1) removed, or
  2) covered by tile(s) on higher layer with XY overlap, or
  3) blocked in all four same-layer directions (left, right, top, bottom)

Critical caveat:
- ClassicMahjongRule.IsTileFreeWithTopBlockerIgnored currently returns false immediately.
- Treat this function as effectively disabled unless fixed.

Post-match behavior:
- On match, free FaceDown tiles are flipped FaceUp.
- EventManager emits tile and combo events.

Lose and win conditions:
- Win when all required non-time mission progress is completed.
- Lose on:
  - timer reaches zero (LoseReason.TimeUp)
  - no available matches (LoseReason.AddMoves)

========================================
5) Input Flow (Tap + Drag)
========================================

Input stack:
- InputController (polls legacy UnityEngine.Input for touch/mouse)
- GameplayPointerSession (orchestrates pointer lifecycle)
- TapTileInteractor + DragTileInteractor

Tap flow:
- PointerDown raycasts tile.
- Normal click => SelectLogic.OnTileClicked(tile)
- Selected tile click can be deferred to allow drag intent.

Drag flow:
- Drag starts after threshold.
- Drag plane is resolved above board depth.
- Drop handling:
  - if valid matching target touched => forward to SelectLogic match path
  - else restore origin and play invalid-drop feedback

Booster input interception:
- Hammer mode installs one-shot click interceptor through InputController.

========================================
6) Booster Flow
========================================

Controller:
- BoosterController owns button entry points and active booster mode lock.

Unlock levels:
- MagicWand: level 3
- Shuffle: level 5
- Hint: level 7
- Hammer: level 9

Booster behavior:
- Hint:
  - Uses solver path (advanced hint), caches pending hint pair.
- Shuffle:
  - Plays shuffle animations
  - Reassigns tile display data
  - retries for winnable layout (LevelValidation)
- Hammer:
  - user picks one tile, auto-finds matching free tile and matches pair
- MagicWand:
  - computes solution path and auto-matches up to N pairs

Revive integration:
- AddMoves lose revive grants +1 shuffle and requests auto-shuffle.
- TimeUp lose revive adds +60 seconds.

========================================
7) UI, Panels, and Event Bus
========================================

Event backbone:
- EventManager is a static event bus used across runtime.

Panel system:
- PanelManager loads panel prefabs through Addressables:
  - Assets/<ModuleName>/UI/Panels/<PanelName>.prefab (current: Assets/Mahjong/UI/Panels/<PanelName>.prefab)
- Core panels:
  - SettingsPanel, RestartPanel, LosePanel, WinPanel

Gameplay HUD:
- GameplayUI listens to level/time/combo/mission/booster events and updates UI.
- GameplayUiEntrance + TileEntranceMotion coordinate reveal timing.
- Booster unlock intro animation starts after UI entrance completion.

Tutorial hand:
- TutorialHandIndicator is active only during tutorial mode and waits for tile entrance completion.

========================================
8) Data and Persistence Contracts
========================================

Level source:
- MessagePack files: Assets/<ModuleName>/Databases/Levels/Level_*.bytes (current project: Assets/Mahjong/Databases/Levels/Level_*.bytes, currently 1..20).

Serialization:
- LevelData, LayerData, TileData and MissionData use MessagePack with union polymorphism.

Player data:
- PlayerData stored in PlayerPrefs via PlayerDataPrefsStore.

Gameplay session data:
- Runtime snapshot stored as JSON for resume support.

========================================
11) Code Style and Engineering Rules
========================================

Dependency handling:
1. Assume [SerializeField] references are assigned in Inspector.
2. Do not add GetComponent fallback for serialized dependencies unless explicitly required.

Null handling:
1. Do not add unnecessary defensive null checks.
2. Fix root cause (boot order, wiring, contract) instead of masking with early returns.

Animation and async:
1. Prefer LitMotion for motion/tween logic.
2. Prefer UniTask for async flows (avoid new coroutine-based flows).
3. Bind motion lifecycle to owner when needed (AddTo(this)).

Architecture:
1. Keep GameBootstrap as scene entrypoint/orchestrator, not gameplay brain.
2. Keep Core models as data contracts.
3. Keep runtime mutable state in runtime classes (BoardState, GameContext, tile runtime states).
4. Respect existing event-driven boundaries (EventManager).

Consistency:
1. Match existing naming and folder boundaries before introducing new abstractions.
2. Keep edits scoped; avoid unrelated refactors.

========================================
12) AI Change Checklist (Mandatory)
========================================

Before implementing, identify impacted contracts:

- If changing match/free-tile logic:
  - ClassicMahjongRule
  - SelectLogic
  - WinPossibilityChecker
  - Booster solver paths (Booster + LevelValidation)

- If adding/modifying tile type:
  - TileData union types
  - TileKind + board spawn + runtime snapshot conversion
  - Editor TileView serialization/deserialization

- If adding/modifying mission type:
  - MissionData union types
  - LevelSerializer build/load mission paths
  - StartedState mission progression
  - MissionUI presentation
  - Editor mission entry tooling

- If changing booster behavior:
  - BoosterId + BoosterUnlockLevel
  - BoosterController
  - Booster UI slots and counts
  - PlayerData booster count persistence

- If changing scene/startup flow:
  - LoadingBar
  - GameBootstrap
  - GameplayUiEntrance / TileEntranceMotion sync events

Always verify:
1. Event subscribe/unsubscribe symmetry.
2. Win/Lose/Pause/Revive transitions still valid.
3. Level serialization compatibility (MessagePack contracts).
